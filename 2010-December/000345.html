On Wed, Dec 1, 2010 at 10:53 AM, David Paulsen <djpdude at gmail.com> wrote:
> Greetings all.  I've recently joined a project wherein a significant portion
> of application input comes from a TS device via what looks to be quite
> typical use of TSLIB: ts_open(), ts_config(), loop on ts_read().
> In times past, I've built test scripts and automation based upon things like
> shell scripts, Expect (http://www.nist.gov/el/msid/expect.cfm
>
> ), python scripts driving Windows applications via their COM interfaces, and
> the like.
> I'd like to bring a similar level of test script-ability to this project's
> TSLIB interface, doing things like writing a set of struct ts_sample to
> *someplace* where ts_read() would pick them up just as if they had come from
> the actual TSDEVICE.
> Brainstorming and hacking, I first tried using a named pipe, which ts_open()
> didn't care for much, an then a Linux guru down the hall suggested a
> pseudo-tty, which seems to have some potential.  (Alternatively, I'm looking
> at somewhat unattractive alterations to the code - things like,
> "if(in-test-script-mode) read(fd...); else ts_read(ts...);"  Yuck.)

No,. you should never have to resort to if's for this, even if a
pseudo tty doesn't work  Absolute worst case you could create a custom
raw module to deliver events from whichever fake source you desire,
rather than using input_raw.
